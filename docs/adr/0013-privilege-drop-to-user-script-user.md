# Drop privileges to the User Script user after privileged setup

`bunsen-core` does its genuinely-privileged setup as root, then performs a one-way drop to the **User Script user** for the rest of its lifecycle. The model is single-process, Path-1 privilege separation: privileges drop **early** — before any persistent state is created — a single retained-privilege worker thread performs the handful of network operations and the privileged DNS bind, and Firecracker plus the Workspace ext4 read run as the User Script user. **No `CAP_SYS_ADMIN` is required anywhere.** This decision supersedes the mount-based *extraction mechanic* of [ADR-0011](0011-hardened-git-fetch-from-sandbox.md) while preserving that ADR's hardened-`git-fetch` posture intact.

## Why

Run under `sudo` — which bunsen needs, to create the per-Run TAP device, load the L3 nftables ruleset, probe the host firewall, and (previously) mount the Workspace ext4 — `bunsen-core` treated root as "the current user". Two consequences followed:

- All persistent state (Sessions, the [[branch-pool]], Run transcripts, agent history, and the kernel/rootfs caches) was written under `/root` and owned by root, so the operator could not find, inspect, or `git worktree add` into it as their normal user.
- `bunsen-core` stayed root for the *entire* Run — including the long-lived phase that relays the Coding Agent's vsock stdio, runs the L7 egress proxy, and serves DNS. That is precisely the surface handling adversarial-agent-influenced bytes, and it ran as root on the host.

The file-ownership defect is caused by state directories being created *before* networking is ever touched. Dropping before that work means all persistent state and caches are created as the User Script user with no `chown` step. The privileged network operations that occur later (and at teardown) therefore cannot rely on the whole process being root — hence a small, retained capability set held by one worker thread rather than the process.

This is also coherent with [ADR-0001](0001-firecracker-as-v1-sandbox-provider.md)'s threat model. The project spends a microVM kernel boundary's worth of capital resisting an adversarial agent; running the host-side code that handles the agent's bytes as root would undercut that. ADR-0005 made the analogous "don't trust a single layer" argument for egress.

## Considered Options

- **Stay root for the whole Run** (the status quo). Rejected: state lands under `/root`, and the agent-facing proxy/DNS/vsock surface runs as root for the Run's entire long-lived phase — the largest, longest-lived privileged surface in the system, handling the least-trusted bytes.
- **Path 1 — single-process drop with a retained-privilege worker thread** (chosen). One `bunsen-core` process starts privileged, drops early, and keeps a narrow capability set on a single OS thread that is the *only* code permitted to run privileged commands. Smallest moving parts; no IPC. Cost: the capability/uid hand-off across threads is subtle (see Consequences) and is the riskiest part of the implementation.
- **Path 2 — separate privileged helper process.** A helper binary holds the privilege; the main process is never root after setup. Cleaner capability isolation, but a whole helper binary plus an IPC protocol to build and secure. Kept as the documented fallback if the in-process capability hand-off proves unworkable in practice; not built here.
- **Userspace `debugfs` read vs kernel `mount` for extraction** (chosen: `debugfs`). Reading the Workspace ext4 with `debugfs -R "rdump …"` needs no `CAP_SYS_ADMIN`; mounting does. It also *improves* the adversarial-filesystem posture: a malformed or hostile ext4 is parsed by an unprivileged userspace tool (blast radius = the User Script user account) instead of by the in-kernel ext4 driver under `CAP_SYS_ADMIN`. `debugfs` ships with `e2fsprogs`, already required because the OCI→ext4 conversion uses `mkfs.ext4`.

## Consequences

- **The drop is early, before state I/O.** Resolution of the target account and the drop happen before any state directory or cache is created, so Sessions, the Branch Pool, transcripts, agent history, and the kernel/rootfs caches are owned by the User Script user regardless of `sudo`. The User Script user is resolved from the invocation context — `--as-user`, the `SUDO_*` environment, or the real uid — never from the `sudo`-mangled inherited `HOME`/`XDG_*`.

- **Per-process drop; no long-lived daemon.** Every operation is its own short-lived `bunsen-core` process: one per Run (`Session.run` / `run_sync`) and one per Session verb (`open`/`close`/`show`/`list`/…). The drop is one-way *within* a process; driving many Runs needs no "un-drop" because each Run is a fresh process that starts privileged → sets up → drops → runs → tears down → exits. Nothing re-escalates after a drop. Session verbs resolve and drop too (their state is user-owned) but perform no network setup, so they never construct the privileged worker.

- **Two supported launcher models.** What stays privileged across many Runs is the launcher, of which there are exactly two: (a) **`sudo` the orchestrator** — the Python process is root and spawns each Run's `bunsen-core` as root (resolution via `SUDO_*`); the User Script itself runs privileged for the whole session. (b) **setuid / file-caps `bunsen-core`** — the Python orchestrator stays unprivileged and each Run's core escalates per-invocation then drops (resolution via the `euid==0 && real-uid!=0` branch). Both are first-class in the resolution order; the setuid/caps *install mechanism* itself is out of scope.

- **A single retained-privilege worker holds the only post-drop capabilities.** A dedicated OS thread (the PrivilegedNet actor) is the sole holder of `CAP_NET_ADMIN`, `CAP_NET_RAW`, `CAP_NET_BIND_SERVICE`, and `CAP_SYSLOG` — and never `CAP_SYS_ADMIN`. The rest of the program reaches it through a small command interface (probe firewall, add/remove TAP allow, create/delete TAP, apply/delete nft, bind DNS, spawn `journalctl`) rather than shelling out to privileged commands directly. It spawns privileged children **synchronously on its own thread** so they inherit its ambient capabilities; async spawning is avoided because the runtime may fork/exec on an arbitrary thread that would not carry them. The TAP is created **owned by the target user** (`ip tuntap add … user <target>`) so the capability-less, post-drop Firecracker can open and attach it.

- **The privileged `:53` DNS bind is done while the capability is still held, then handed off.** The actor performs the bind and returns the bound socket as a file descriptor, which the ordinary asynchronous DNS listener adopts. DNS works without retaining `CAP_NET_BIND_SERVICE` anywhere except the actor. Likewise `journalctl -k` (the L3 drop-log source) is spawned by the actor so it inherits `CAP_SYSLOG`, keeping the drop-log working regardless of the target user's journal-group membership.

- **Firecracker runs as the User Script user.** It is launched *after* the drop, so its API socket, vsock socket, and working directory are owned by the User Script user — the dropped supervisor connects to and cleans them up without privilege. This requires the target user to be able to reach `/dev/kvm` (the `kvm` group).

- **Extraction is userspace and supersedes the ADR-0011 mount.** The `losetup` + `mount` + `umount` extraction is replaced by a `debugfs -R "rdump …"` read of the Workspace ext4 into a user-owned temporary directory, with symlinks scrubbed to preserve the ADR-0011 anti-exfiltration posture. The **hardened `git fetch`** defined by ADR-0011 — its flag set, deprivileged execution, explicit refspec, and the narrow agent-history copy — is unchanged and runs against the extracted `.git`. The extraction interface (Workspace ext4 + Run id + Pool + optional output branch + adapter + history destination → Pool refs) is unchanged, so callers and the User Script workflow are unaffected.

- **Fail-closed when root with no resolvable target.** If `bunsen-core` is root but cannot determine which user to become (no `--as-user`, no `sudo` context), it refuses to run with an actionable error rather than silently writing root-owned state. The resolved uid must be non-zero (never "drop to root") and the account's home must resolve and exist. A normal-user invocation with no elevation attempts no drop, preserving today's dev path.

- **The capability/uid sequencing is load-bearing and is the riskiest part.** Set keep-capabilities-across-uid-change on the worker thread and populate its capability sets *first*; then on the main thread `initgroups → setresgid → setresuid` to the target. The C library broadcasts the uid change to every thread, and each thread honours its own keep-capabilities flag, so the worker retains its caps while every other thread becomes capability-less. The tokio runtime is constructed *after* the drop so its workers start unprivileged. If a clean in-process hand-off cannot be achieved, fall back to the Path-2 helper rather than weakening the drop — in particular, do not retain capabilities process-wide, which would leak them into `firecracker`, `git`, and `debugfs`.

- **No Python change.** The wrapper already inherits the process environment when spawning `bunsen-core`, so `SUDO_*` reach the core. Adopting this is purely operational: run `sudo python <script>`, and ensure the target user is in the `kvm` group and can read the kernel log.

This decision pairs with [ADR-0011](0011-hardened-git-fetch-from-sandbox.md) (the fetch security posture, preserved) and [ADR-0010](0010-session-and-branch-pool.md) (the Pool that extraction writes into).
